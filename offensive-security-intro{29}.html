01 · Resumen Ejecutivo

Este room introductorio de TryHackMe simula el compromiso de una aplicación bancaria ficticia llamada FakeBank. El objetivo es experimentar de primera mano qué significa la seguridad ofensiva: pensar como un atacante para identificar fallos antes de que los actores maliciosos lo hagan.

Utilizando un ataque de fuerza bruta de directorios con la herramienta dirb, se descubrió un panel de transferencias bancarias (/bank-transfer) completamente desprotegido. Sin ningún tipo de autenticación, fue posible realizar un depósito fraudulento en la cuenta objetivo #8881, haciendo positivo su saldo y revelando la flag del laboratorio.

Contexto
El laboratorio corre en un entorno web aislado dentro de TryHackMe. La terminal web tiene dirb preinstalado. No se requirieron herramientas externas.

03 · Fases de Ataque

01
Recon Reconocimiento Inicial — Inspección de FakeBank

Se cargó el laboratorio usando el botón "View Site" de TryHackMe, que abrió la interfaz de FakeBank en el navegador integrado. Se procedió a una inspección visual sistemática de la aplicación.

La landing page de FakeBank mostraba una lista de cuentas bancarias con sus saldos actuales. Visualmente, la aplicación imita una banca en línea legítima, con secciones de transferencia y consulta de saldo. Sin embargo, a primera vista carece de elementos de seguridad visibles como formularios de login, tokens CSRF o restricciones de navegación.

🎯
URL base identificada
http://fakebank.thm — dominio interno del laboratorio
💳
Cuenta objetivo
Número #8881 visible en la interfaz principal sin autenticación previa
⚠️
Hipótesis inicial
La ausencia de login implica que funciones administrativas pueden estar expuestas en rutas no enlazadas. Candidato a fuerza bruta de directorios.
02
Discovery Directory Bruteforce con dirb

Con la URL base identificada, se ejecutó dirb desde la terminal web integrada del laboratorio. Esta herramienta realiza fuerza bruta sobre rutas del servidor usando un diccionario de palabras comunes para descubrir recursos ocultos.

$ dirb http://fakebank.thm ----------------- DIRB v2.22 By The Dark Raver ----------------- START_TIME: Thu Apr 30 2026 URL_BASE: http://fakebank.thm/ WORDLIST_FILES: /usr/share/dirb/wordlists/common.txt ----------------- GENERATED WORDS: 4612 ---- Scanning URL: http://fakebank.thm/ ---- + http://fakebank.thm/images (CODE:301|SIZE:315) + http://fakebank.thm/bank-transfer (CODE:200|SIZE:2048) ----------------- END_TIME: ... DOWNLOADED: 4612 - FOUND: 2 -----------------
📁
Ruta 1: /images
Directorio de recursos estáticos. Código 301 — redirección. Baja relevancia ofensiva.
🔓
Ruta 2: /bank-transfer — HALLAZGO CRÍTICO
Panel de transferencias bancarias accesible con código 200 OK sin ningún tipo de autenticación ni token de sesión. Vector de explotación confirmado.

05 · Vulnerabilidades Identificadas

VULN-01 Panel administrativo oculto sin autenticación Crítica
Descripción El endpoint /bank-transfer no requiere ningún tipo de login, sesión válida ni token de autorización. Es accesible directamente por cualquier cliente HTTP.
Impacto Un atacante puede depositar o retirar fondos de cualquier cuenta del banco sin restricción alguna, comprometiendo la integridad financiera de toda la aplicación.
CWE CWE-306 — Missing Authentication for Critical Function
Mitigación Implementar middleware de autenticación obligatorio en todas las rutas de gestión financiera. Aplicar control de acceso basado en roles (RBAC): solo usuarios con perfil "admin" o "staff" deben acceder a transferencias.
VULN-02 Exposición de directorios internos por directory listing / rutas predecibles Baja
Descripción dirb descubrió en segundos dos rutas internas (/images y /bank-transfer) usando únicamente un diccionario de palabras comunes.
Impacto Facilita la fase de Discovery del atacante, reduciendo el tiempo necesario para identificar superficies de ataque ocultas.
CWE CWE-548 — Exposure of Information Through Directory Listing
Mitigación Deshabilitar directory listing en la configuración del servidor web. Usar rutas con tokens aleatorios o segmentos no predecibles para funcionalidades sensibles.
VULN-03 Validación insuficiente de transacciones financieras Alta
Descripción El formulario de depósito acepta cualquier valor positivo sin verificar límites, disponibilidad de fondos de origen, autenticación del solicitante ni autorización secundaria.
Impacto Manipulación arbitraria de saldos bancarios. En un sistema real equivaldría a fraude financiero directo.
CWE CWE-20 — Improper Input Validation
Mitigación Validar montos server-side, establecer límites diarios, verificar fondos disponibles en la cuenta origen, implementar autorización dual (4-eyes principle) para transacciones superiores a cierto umbral.

07 · Lecciones Aprendidas

01
La seguridad ofensiva es pensar como el atacante. Identificar los mismos fallos que explotaría un actor malicioso, pero en un entorno controlado y con autorización explícita, es la esencia del ethical hacking.
02
Los endpoints ocultos no son endpoints seguros. Si una ruta no está enlazada en la UI pero existe en el servidor, herramientas como dirb la encontrarán en minutos. La seguridad por oscuridad no es una estrategia válida.
03
La autenticación es la primera línea de defensa. Cualquier funcionalidad que pueda modificar datos sensibles debe estar protegida por autenticación y autorización, independientemente de si es "visible" para el usuario o no.
04
La validación server-side es no negociable. Nunca confiar en el cliente para validar montos, permisos o identidad. Toda la lógica de negocio sensible debe validarse en el backend.

08 · Conclusión

Offensive Security Intro cumple su propósito pedagógico con efectividad: en minutos demuestra que incluso un atacante con conocimientos básicos puede comprometer un sistema si los controles de acceso fundamentales no están en su lugar. La cadena de ataque — recon visual → fuerza bruta de directorios → acceso sin autenticación → modificación de datos — es representativa de vulnerabilidades reales que aparecen en aplicaciones web de producción.
